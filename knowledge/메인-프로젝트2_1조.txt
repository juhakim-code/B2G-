[2026-01-21 04:12] 0) 큰 그림
1. *Cloud SQL(PostgreSQL) 인스턴스 생성*
*DB/유저/권한 세팅*
1. (선택) *IAM DB 인증* 또는 그냥 *DB 유저/비번*
2. *로컬에서 Postgres 접속 확인* (Cloud SQL Auth Proxy)
3. *SQLite → Postgres 마이그레이션(스키마 + 데이터)*
4. *앱 코드의 DB 연결을 Postgres로 변경*
5. *Cloud Run 배포 + Cloud SQL 연결 추가*
6. *배포 시 마이그레이션 자동 실행*(권장)
7. *운영 체크(백업/모니터링/커넥션 풀)*
1) Cloud SQL(PostgreSQL) 만들기 (콘솔)
1-1. 인스턴스 생성
• GCP 콘솔 → *Cloud SQL → 인스턴스 만들기(Create instance) → PostgreSQL 선택*
 이 흐름이 공식 문서에 그대로 나와 있어.
*추천 설정(초기 MVP용, 비용 절감 목적)*
• Region: 앱이 돌 지역이랑 동일(예: asia-northeast3 서울)
• 머신 타입: 최소 사양부터 시작 (사용량 보고 올리기)
• 스토리지: 자동 증가 켜기(나중에 꽉 차서 장애 나는 거 방지)
• 고가용성(HA): MVP면 처음엔 꺼도 됨(비용↑)
&gt; 주의: DB 패스워드는 안전하게 따로 저장(비번 분실하면 귀찮아짐)
2) DB/계정/권한 세팅
2-1. DB 생성
• Cloud SQL 인스턴스 → Databases → *Create database*로 앱 전용 DB 생성
 (ex: `uniwerse_prod`)
2-2. 앱 전용 유저 생성
• Users → *Create user*
 (ex: `app_user`)
• 권한은 “해당 DB에 필요한 범위만” 주는 게 좋아.
&gt; 운영에서는 `postgres` 슈퍼유저를 앱에서 직접 쓰는 건 보통 피함.
3) (강추) 로컬 개발/마이그레이션용 접속: Cloud SQL Auth Proxy
SQLite에서 Postgres로 옮길 때, 로컬 PC에서 Postgres에 안정적으로 붙는 게 먼저야.
• GCP 공식: *Cloud SQL Auth Proxy로 연결*하는 방법이 문서로 제공돼. 
• 인스턴스 “Connection name(project:region:instance)”도 문서대로 복사. 
*흐름*
로컬에 Auth Proxy 실행
1. 로컬에서 `psql`/툴로 `localhost:5432` 등에 접속
2. DB/테이블 확인
4) SQLite → Postgres 마이그레이션(스키마 + 데이터)
여기가 제일 중요하고, 실수 많이 나는 구간이야.
4-1. 스키마(테이블 구조)부터 결정
• 이미 ORM(Prisma/TypeORM/Django 등) 있으면:
ORM의 DB 설정을 Postgres로 바꿈
*마이그레이션 실행해서 Postgres에 테이블 생성*
• ORM 없으면:
• SQLite의 테이블 구조를 기준으로 Postgres용 `CREATE TABLE`을 만들어야 함
 (SQLite는 타입이 느슨해서, Postgres로 옮길 때 타입 정리가 필수)
*자주 터지는 타입 이슈*
SQLite의 날짜 TEXT → Postgres `timestamp`/`timestamptz`
0/1 값 → Postgres `boolean`
• autoincrement → Postgres `serial`/`identity` + sequence 정리
4-2. 데이터 이관(가장 흔한 방식)
테이블별로 SQLite에서 CSV로 뽑아서
• Postgres에서 `COPY`로 넣는 방식이 제일 단순하고 디버깅이 쉬워.
&gt; “한 방에 자동 이관” 도구도 있지만(추측), 앱 구조/타입에 따라 깨질 때가 많아서
&gt;  언니처럼 서비스 앱이면 *테이블별로 통제하면서 넣는 게 안전*해.
5) 앱 코드에서 SQLite → Postgres로 교체
5-1. 절대 원칙
• *프론트(React)에서 Postgres 직접 접속 금지.*
• Postgres는 **백엔드(Cloud Run 같은 서버)**에서만 붙어야 해.
즉, 언니 프로젝트가 “프론트만 있는 구조”였다면
 → 이 시점에 *백엔드 API*가 필요해져.

6) Cloud Run 배포 + Cloud SQL 연결 (추천 루트)
Cloud Run은 “크레딧 활용하면서 관리 부담 적게” 가기 좋아.
6-1. Cloud Run 서비스에 Cloud SQL 연결 추가
GCP 공식 문서 순서가 딱 이거야:
 Cloud Run 서비스 → Edit &amp; deploy new revision → Container settings → *Cloud SQL connections에서 Add connection*.
6-2. 권한(서비스 계정) 체크
• Cloud Run에서 쓰는 서비스 계정에 *Cloud SQL Client* 권한이 필요해.
 (이거 안 주면 연결 실패 많이 남)
6-3. 환경변수 설정
Cloud Run에 아래를 환경변수로 넣어:
• `DB_NAME`, `DB_USER`, `DB_PASS` (또는 IAM 인증 시 다른 방식)
`INSTANCE_CONNECTION_NAME` (project:region:instance)
• 프레임워크에 맞는 `DATABASE_URL`
7) 운영 안정화: IAM DB 인증(선택이지만 추천)
비번을 코드/환경변수로 들고 다니기 싫으면 *IAM DB 인증*이 좋아.
• Cloud SQL for PostgreSQL은 **IAM DB 인증(automatic/manual)**을 지원하고, “자동(커넥터/프록시가 토큰 갱신)”을 더 추천한다고 문서에 명시돼 있어. 
• 장점: 비번 회전/유출 리스크 줄어듦
• 단점: 초기 셋업이 살짝 더 복잡
8) 배포 “수시 업데이트 + 저장”에서 중요한 3가지
1. *마이그레이션 자동화*
Cloud Run 새 리비전 배포할 때 DB 스키마도 같이 최신화되게
2. *커넥션 관리(풀링)*
서버리스 성격이면 커넥션 폭주로 DB가 버벅일 수 있음(언니 사용량 늘면 특히)
3. *백업/복구 전략*
Cloud SQL는 백업/포인트인타임 복구(PITR) 같은 기능을 켜는 쪽이 안전

9) 언니가 “지금 당장” 따라할 체크리스트 (복붙용)
•  Cloud SQL PostgreSQL 인스턴스 생성 
•  DB 생성 + 앱 유저 생성
•  로컬에서 Cloud SQL Auth Proxy로 접속 확인 
•  Postgres에 스키마 생성(ORM 마이그레이션 or 수동)
•  SQLite → CSV export → Postgres COPY import
•  백엔드 DB 드라이버를 Postgres로 변경
•  Cloud Run 배포 후 Cloud SQL 연결 추가 
•  환경변수(DATABASE_URL 등) 세팅
•  (선택) IAM DB 인증 적용
